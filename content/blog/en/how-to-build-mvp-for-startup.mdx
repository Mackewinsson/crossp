Most startups do not fail because they cannot write code. They fail because they spend months building features nobody asked for. An MVP should buy learning — not decorate a pitch deck.

## Start with one job-to-be-done

Write one sentence: *For [user], my product helps them [job] so they can [outcome].* If you need three sentences, you have three products. Pick one for v1.

## Cut ruthlessly

Keep in the first version:
- authentication (only if required for the job),
- the core workflow end to end,
- a simple admin path so you are not stuck in the database,
- production deploy with staging.

Cut for later:
- multi-role complexity,
- every integration “we will need someday,”
- beautiful edge-case polish,
- analytics dashboards nobody will open in week one.

## Build sequence that protects runway

1. Scope the shippable slice in writing.
2. Ship weekly demos of working software.
3. Put it in front of real users before expanding.
4. Only then fund the next slice.

## Common MVP traps

- Confusing a Figma prototype with a product.
- Hiring for speed and skipping ownership of the repository.
- Asking for a 6-month fixed quote before you have talked to users.

> An MVP that cannot be used by a real customer is a science project with branding.
